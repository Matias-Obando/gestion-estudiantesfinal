Introducción

En esta parte de la práctica trabajaremos con bases de datos mediante JDBC para evitar que la información de la aplicación se pierda cada vez que el programa finaliza. Este proceso forma parte de la gestión de la persistencia, es decir, del almacenamiento duradero de los datos. Aunque los ficheros también proporcionan persistencia, en la industria las bases de datos son la opción más habitual y la que utilizaremos aquí.
Es importante remarcar que, aunque utilicemos una base de datos, seguiremos necesitando estructuras de datos en memoria como List, Set o Map. Estas estructuras permiten que la aplicación funcione de manera ágil y eficiente. Por tanto, el acceso a la base de datos debería realizarse solo en momentos concretos:
•	al inicio del programa, para cargar los datos, y
•	al final, para guardar los cambios realizados.
Acceder continuamente a la base de datos dentro del flujo normal del programa puede ser costoso y provocar cuellos de botella. Por este motivo, durante la práctica trabajaremos con datos en memoria y solo consultaremos la base de datos cuando resulte necesario.

Práctica 6 – Sesiones HTTP

Primera parte									
Introducción

En esta práctica vamos a introducir el manejo de sesiones y el control de acceso para determinadas operaciones de nuestro sitio web. Esto proporcionará a nuestro proyecto una mayor funcionalidad.

session vs. request 

Una sesión la podemos ver como una “tabla de dos columnas” que el servlet mantiene en memoria. Una columna sirve para guardar el nombre de una variable (atributo) y la otra para guardar su valor, como por ejemplo: 

ATRIBUTO	VALOR
usuario	Antonio
edad	40

Dado que el sevlet mantiene en memoria la “tabla” esto nos permite guardar y recuperar valores de ella. Podemos ver como la filosofía de las sesiones (session) es muy parecida a la de las peticiones (request) con las que ya hemos tratado un poco en ejercicios anteriores. Pero hay diferencias. A saber: 

En una petición (requets) la “tabla” de información sólo está disponible en el espacio de tiempo entre que el servlet lee valores que el usuario le pasa desde el navegador (por ejemplo, cuando le pasa su nombre y edad desde un formulario) o al revés, cuando el usuario lee valores  que el servlet le ha dejado preparados (por ejemplo, una lista con todos los alumnos existentes).

En cambio, en una sesión la información de la “tabla” está siempre disponible para cualquier usuario y desde cualquier página de nuestro sitio web en tanto que no se elimine la sesión. La eliminación de la sesión puede ser provocada por el propio usuario (por ejemplo, al pulsar un botón para desconectar) o por el propio servlet, si se ha alcanzado un determinado tiempo de inactividad por parte del usuario.


De las clases de teoría sabemos que cuando un usuario se conecta a un servidor web se crea una sesión que lo identifica unívocamente (quiera o no el usuario). Ahora bien, el programador puede utilizar esta información subyacente a la sesión que se establece entre el navegador y el servidor web o puede decidir no utilizarla.

Enunciado de la práctica

De momento cualquier usuario de nuestro sitio web puede hacer cualquier operación: consultar los alumnos existentes, dar de alta alumnos y visualizar listados de alumnos.  Además, desconocemos totalmente quienes son los usuarios que acceden a nuestro sitio web: no sabemos su nombre, ni en que momento han accedido, ni durante cuanto tiempo han estado navegando, etc.

Esta situación tiene que cambiar. Primero vamos a implementar un registro de información sobre la sesión del usuario y después el control de acceso a determinadas operaciones.

¿Cómo se hace lo anterior?

En el método service del servlet hay que poner la instrucción que crea o utiliza una sesión. El parámetro ‘true’ indica crear la sesión si esta no existe.

// Creación de la sesión. Si ya existe la utiliza			
HttpSession sesion = request.getSession(true);

Luego hay que llamar a un método local encargado de incrementar el contador de veces que el usuario ha invocado al servlet:

incrementarContadorSesion(sesion);

A continuación el código del método:

/*
 * Metodo para llevar el control del numero de veces
 * que accede al servlet un usuario
 */
private void incrementarContadorSesion(HttpSession sesion) {
			
	// Variable para guardar las veces que accede el usuario
	Integer contadorAccesos=new Integer(0);
		
	// Si ya existe la sesion...
	if (!sesion.isNew()) {
		// Recuperar el contador
		Integer contadorActual=(Integer)sesion
.getAttribute("contadorAccesos");
		// Incrementarlo
		if (contadorActual!=null) {
			contadorAccesos=
				new Integer(contadorActual.intValue()+1);
		}	
	}
	
	// Guardar el nuevo valor del contador
	sesion.setAttribute("contadorAccesos", contadorAccesos);			
}

Gestión de eventos de sesión en aplicaciones web

Introducción

Existen determinadas situaciones en proyectos Web en las que necesitamos saber cuando se ha creado/destruido una sesión o cuando se ha añadido/eliminado un objeto de ella. Con este ejercicio practicaremos con este tipo de operaciones. 
Nuestro objetivo va a ser añadir más información a la página que muestra los datos de la sesión. Queremos mostrar una página parecida a esta:

 
Las dos últimas filas indican que hay un total de dos usuarios conectados a nuestro sitio web pero sólo uno (daniel) ha sido validado (añadido en la sesión).

Para obtener los valores de las dos últimas filas nuestra aplicación se tendrá que encargar de mantener un contador del número de usuarios que actualmente se han conectado a nuestra aplicación y otro contador para controlar cuántos de ellos se han logado.

¿Cómo lo haremos?
Primero veamos algo de teoría sobre las siguientes interfaces:
 
HttpSessionListener y HttpSessionBindingListener (javax.servlet.http)

HttpSessionListener y HttpSessionBindingListener son dos interfaces del paquete javax.servlet.http. Las clases que las implementen tendrán un comportamiento especial en nuestro contenedor de servlets (Tomcat). 

HttpSessionListener define dos métodos: sessionCreated() y sessionDestroyed(). Estos métodos son utilizados por el contenedor de aplicaciones para notificarnos cuando la sesión de un usuario ha sido, respectivamente, creada o destruida. Aquellas clases que implementen este interfaz deberán ser registradas como listeners en nuestro fichero web.xml (en el ejemplo que se muestra más abajo la implementa la clase webses.WebSessionListener). 

Por otro lado, HttpSessionBindingListener, contiene dos métodos: valueBound() y valueUnBound(), invocados por el contenedor para indicar a la clase que implemente esta interfaz que una instancia de su propia clase ha sido insertada o eliminada de la sesión. En nuestro caso esto sucederá cuando el usuario haya hecho login y logout, respectivamente).

En nuestro ejemplo la clase beans.Usuario implementa HttpSessionBindingListener. Así, cuando hagamos: 
sesion.setAttribute("usuario", usuario);
Tomcat llamará al método valueBound().
 
ServletContext (el contexto de la aplicación)

Los contadores que mantienen el número de usuarios conectados y logados deben estar almacenados en el Contexto del contenedor y cualquiera que intente modificarlos u obtener su valor deberá utilizar una región crítica (parte de código que no puede ser ejecutada por más de un hilo simultáneamente). Esto es necesario porque pueden darse situaciones donde varios hilos en paralelo puedan estar operando con el mismo código (con nuestros contadores), lo que puede provocar inconsistencias. 

¿Qué es el contexto del contenedor (o de la aplicación web)?

El ServletContext permite manejar propiedades y métodos del contexto de los servlets, que viene a ser un espacio común para todos los servlets de nuestra aplicación. Por lo tanto todos los servlets dentro de una misma aplicación web tienen el mismo ServletContext, por eso éste gestiona atributos que se cargan a nivel de toda la aplicación web. Es lo que se llama en otras tecnologías variables de aplicación o globales para el caso de aplicaciones web, ya que todos (jsp y servlets) podrán ver y compartir estos mismos atributos.

Como hemos comentado antes, tenemos que crear la clase WebSessionListener.java encargada de mantener el contador del número de usuarios que se encuentran conectados a nuestra aplicación en un momento dado. El método sessionCreated() aumentará el contador "usuariosConectados", mientras  que sessionDestroyed() lo disminuirá:

package webses;

import javax.servlet.ServletContext;
import javax.servlet.http.HttpSessionEvent;
import javax.servlet.http.HttpSessionListener;

public class WebSessionListener implements HttpSessionListener {

public void sessionCreated(HttpSessionEvent arg0) {

System.out.println("Session creada");

ServletContext contexto = 
		arg0.getSession().getServletContext();

synchronized (contexto) {

Integer usuariosConectados = 
(Integer) contexto
.getAttribute("usuariosConectados");
if (usuariosConectados == null) {
usuariosConectados = new Integer(0);
}
usuariosConectados+=1;
contexto
.setAttribute("usuariosConectados",
				usuariosConectados); 

} // synchronized

}

public void sessionDestroyed(HttpSessionEvent arg0) {

System.out.println("Session destruida");

arg0.getSession().getServletContext();

Integer usuariosConectados = 
(Integer) contexto
.getAttribute("usuariosConectados");

usuariosConectados = new Integer(0);
}
usuariosConectados-=1;
contexto
.setAttribute("usuariosConectados",
 usuariosConectados); 

} // synchronized

}

}

Por otro lado, tenemos que crear la clase Usuario.java, un bean que almacenará los datos del usuario (de momento nos basta con el nombre del usuario) y que se encargará de incrementar (en el método valueBound()) o decrementar (en el método valueUnBound()) el contador de usuarios logados en la aplicación:

package beans;

import javax.servlet.ServletContext;
import javax.servlet.http.HttpSessionBindingEvent;
import javax.servlet.http.HttpSessionBindingListener;

public class Usuario implements HttpSessionBindingListener {

	private String nombre="";
	
	public Usuario(String nombre) {
		this.nombre=nombre;
	}
	
	public String getNombre() { return this.nombre; }
	
	public void valueBound(HttpSessionBindingEvent arg0) {
		
		System.out.println("User añadido a la session. ID: "+
				arg0.getSession().getId());
		
		ServletContext contexto = arg0.getSession().getServletContext();

		synchronized (contexto) {

			Integer usuariosValidados = (Integer) contexto
					.getAttribute("usuariosValidados");
			if (usuariosValidados == null) {
				usuariosValidados = new Integer(0);
			}
			usuariosValidados += 1;
			contexto.setAttribute("usuariosValidados", usuariosValidados);

		} // synchronized
	}

	public void valueUnbound(HttpSessionBindingEvent arg0) {
		
		System.out.println("User eliminado de session. ID: "+
				arg0.getSession().getId());
		
		ServletContext contexto = arg0.getSession().getServletContext();

		synchronized (contexto) {
			Integer usuariosValidados = (Integer) contexto
					.getAttribute("usuariosValidados");

			if (usuariosValidados == null) {
				usuariosValidados = new Integer(0);
			}
			usuariosValidados -= 1;
			contexto.setAttribute("usuariosValidados", usuariosValidados);

		} // synchronized
	}
}

Por otro lado, tenemos que actualmente utilizamos un método privado en el servlet Controlador para validar al usuario. Este método lo utiliza el propio servlet Controlador para añadir el nombre del usuario a la sesión en curso. Podríamos mejorar esto: además de las dos clases anteriores (WebSessionListener y Usuario) podríamos crear un servlet llamado Login.java que será llamado desde el servlet Controlador para hacer la operación de validación que hasta el momento estaba haciendo el propio servlet Controlador.

package servlets;

import java.io.IOException;
import javax.servlet.ServletException;
import javax.servlet.http.HttpServlet;
import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;
import beans.Usuario;

public class Login extends HttpServlet {

	private static final long serialVersionUID = -7432700501449176526L;
	private static final String PG_RESPUESTA_ERROR = "/error.jsp";
	private static final String PG_RESPUESTA_ALTA = "/Controlador?operacion=alta";
	private String siguientePag = "";
	
	@Override
	protected void doGet(HttpServletRequest arg0, HttpServletResponse arg1)
			throws ServletException, IOException {

		doPost(arg0, arg1);
	}

	@Override
	protected void doPost(HttpServletRequest arg0, HttpServletResponse arg1)
			throws ServletException, IOException {
		
		String usuarioIntro = arg0.getParameter("txtUsuario");
		String passwIntro = arg0.getParameter("txtContrasenya");
		
		// Intentamos obtener el nombre del usuario si es que está
		// ya validado
		String sessionUser = (String) arg0.
				getSession().getAttribute("nombre");
		
		// Variable que tomará como valor el nombre del usuario
		// confirmado contra la "Base de datos" 
		String user = null;
		
		// Si el usuario no está validado hay que validarlo 
		if (sessionUser == null)
			user = validarUsuario(usuarioIntro, passwIntro);
		
		// Si ni está validado y tampoco se reconoce su usuario-password
		if ((sessionUser == null) && (user == null)) {
			siguientePag=PG_RESPUESTA_ERROR;		
		} else {
			// Si no está validado y el usuario introducido es correcto
			if ((sessionUser == null) && (user!=null)) {
				Usuario usuari=new Usuario(user);
				arg0.getSession().setAttribute("usuario", usuari);				
				arg0.getSession().setAttribute("nombre", usuari.getNombre());
			}
			siguientePag=PG_RESPUESTA_ALTA;
		}
				
		getServletContext().getRequestDispatcher(siguientePag)
				.forward(arg0, arg1);
	}
	
	// Nos devuelve el nombre del usuario si la pareja 
	// nombre-passw es correcta
	// Tendriamos que comparar con valores de la BD!
	private String validarUsuario(String usuarioIntro, String passwIntro) {
		String retorno=null;		
		
		if (usuarioIntro.equals("daniel") && 
				passwIntro.equals("daniel")) {
			retorno=usuarioIntro;
		}
		return retorno;
	}

}
